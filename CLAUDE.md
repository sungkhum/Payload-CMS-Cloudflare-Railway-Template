# Boxcar — context for Claude

This is **Boxcar**, a generic Payload v3 + Next.js blog boilerplate designed to deploy on Railway with Cloudflare R2 for media. It's intended as a reusable starter — anyone forking this should be able to fill in env vars and ship a blog. An optional WordPress migration script is included for users coming from WP.

## Stack

- Payload 3.84 + Next.js 16 (single app: admin at `/admin`, frontend at `/`)
- PostgreSQL on Railway
- Cloudflare R2 for media via `@payloadcms/storage-s3`
- Plugins: `@payloadcms/plugin-seo`, `@payloadcms/plugin-redirects`

## Conventions

- Collection slugs are lowercase plural (`posts`, `pages`, `media`, `categories`, `tags`, `comments`, `users`).
- Posts and Pages have draft/publish via Payload's built-in `versions.drafts`. Public read access is gated on `_status: published`.
- Every public-facing collection (Posts, Pages, Media, Categories, Tags, Comments) carries a `legacy` group field (`wpId`, `wpUrl`) used by the optional WordPress migration script. Users not migrating from WP can leave it empty or strip it from `src/fields/legacy.ts`. If keeping it, don't rename the field names — the migration script and the redirects seeding rely on them.
- Slugs are auto-generated from the title via `src/fields/slug.ts`. The hook is `beforeValidate` and slugifies if the user leaves the slug blank.
- Access helpers live in `src/access/`. Prefer those over inline `({ req }) => …` functions in collection configs.
- Comments allow anonymous create (public can submit). The `beforeChange` hook forces `approved: false` when there's no authenticated user — don't remove this guard.
- Auto-generated Payload files (`src/app/(payload)/admin/[[...segments]]/page.tsx`, etc.) carry a "GENERATED AUTOMATICALLY" header. They're regenerated by `pnpm generate:importmap` and `withPayload` during build — modify with care.
- `src/app/(payload)/admin/importMap.js` and `src/payload-types.ts` are gitignored generated artifacts. Don't commit them. The `postinstall` script auto-generates the importMap on every `pnpm install` (guarded by an `existsSync('src/payload.config.ts')` check so it no-ops during the Docker deps stage where source isn't copied yet — `pnpm build` regenerates it at that point).
- **Scheduled posts: cron-based, no `schedulePublish` flag.** Editors use the existing `publishedAt` date picker — set a future date and click Publish. The Posts `beforeChange` hook detects "publish + future date", demotes `_status` to `draft` so the post stays hidden, and the `afterChange` hook enqueues a `publishScheduledPost` job (defined in `src/lib/tasks/publishScheduledPost.ts`) with `waitUntil = publishedAt`. The `autoRun` cron in `payload.config.ts` polls every minute and the task flips `_status` from `draft` → `published` when its time arrives. The handler is idempotent: it re-checks the doc's current `publishedAt` before flipping, so reschedules and manual publishes don't double-fire. Public reads stay simple (`_status: published`) — no date math at the read site, since the cron keeps `_status` accurate. The `/admin/scheduled` view lists posts with `_status: draft` AND `publishedAt > now()`. The `PublishedAtDescription` field component gives editors live feedback ("Publishes immediately" / "Scheduled — will publish in N days") under the date picker. **Surfaces to smoke-test on Payload upgrades:** `jobs` config shape (tasks, autoRun, shouldAutoRun), `req.payload.jobs.queue` Local API, `admin.components.views` server-component prop shape, `afterNavLinks` slot, field-level `admin.components.Description` slot, `useField` hook from `@payloadcms/ui`, Payload's `--theme-*` CSS variables. The custom views/components deliberately avoid all other `@payloadcms/ui` imports.

## Local dev

```bash
docker compose up -d   # Postgres
pnpm install
pnpm dev               # http://localhost:3125  (admin at /admin)
```

`.env.example` ships with a working local Postgres URI matching `docker-compose.yml`. R2 vars are blank — fill them in to enable media uploads. Default port is 3125 (set via `PORT=3125` in `.env`; honored by `next dev`, `next start`, and the Docker container).

## Deployment

Recommended: connect the GitHub fork to Railway via the Railway GitHub App. **Pushing to `main` then auto-deploys** — do NOT also run `railway up` afterwards or you'll build twice. Reserve `railway up` for emergency deploys that bypass git (e.g. testing un-pushed local changes).

Deploy mechanics:
- Build: `Dockerfile` (multi-stage, full node_modules in runner so the Payload CLI is available — `output: 'standalone'` was deliberately removed).
- Migrations: Railway's `preDeployCommand: "pnpm migrate"` runs them before the new container goes live. A non-zero exit aborts the deploy. Don't combine migrate + start with `&&` in `startCommand` — Railway runs `startCommand` in exec form, so shell operators are not interpreted.
- Start: `pnpm start`, which runs `next start -H 0.0.0.0 -p ${PORT:-3125}`. The `-H 0.0.0.0` is required — without it Next binds to localhost and Railway's healthcheck probe can't reach it.
- Healthcheck: `/healthz` (handler at `src/app/healthz/route.ts`). 120s timeout in `railway.json`.
- `NEXT_PUBLIC_SERVER_URL` and `NEXT_PUBLIC_SITE_NAME` are build-time — change them on Railway and redeploy when DNS cuts over or branding changes.

Railway env vars (set on the app service):
- `DATABASE_URI=${{Postgres.DATABASE_URL}}` (reference, resolved at deploy)
- `PAYLOAD_SECRET`, `NEXT_PUBLIC_SERVER_URL`, `NEXT_PUBLIC_SITE_NAME`, plus all five `R2_*` vars
- Railway auto-injects `PORT`, `RAILWAY_PUBLIC_DOMAIN`, etc.

pnpm version is pinned to `10.32.1` in three places (`packageManager` field in package.json, Dockerfile builder + runner stages) — corepack defaults to pnpm 11 which fails the `engines.pnpm: ^9 || ^10` check. The runner installs pnpm via `npm install -g` rather than corepack to avoid corepack's per-user lazy-download pitfalls.

## Branding

Default brand is **Boxcar** / **boxcar**. The strings live in:
- `src/components/site-header.tsx`, `src/components/site-footer.tsx`
- `src/app/(frontend)/layout.tsx` (metadata)
- `src/app/(frontend)/posts/[slug]/page.tsx`, `pages/[slug]/page.tsx` (per-doc title suffix)
- `src/payload.config.ts` (admin title suffix + SEO `siteName` default)
- `package.json` (`name`, `description`)

The SEO plugin's title generator reads `NEXT_PUBLIC_SITE_NAME` (falling back to `'Boxcar'`) so users can override their site name from env without editing code. The other locations are intended to be search-and-replaced by the user when they fork.

## Optional WordPress migration

WP → Payload migration script lives at `scripts/migrate-wordpress.ts` (stub). When implementing/extending it, populate `legacy.wpId` and `legacy.wpUrl` on every imported document so the Redirects plugin can be seeded from `legacy.wpUrl` → new slug. The source WP site URL should come from an env var (e.g. `WP_SOURCE_URL`) so the boilerplate stays generic.
